⏺ Found it. The OP_CLAIM_ALL opcode in the frontend does not match the compiled
  contract.

  From the ABI: ClaimAll header = 4192066358 = 0xF9DDDB36
  Frontend has: 0xf9e43eb6 (wrong)

  The other three opcodes (CreateChallenge, AddFunds, RefundUnclaimed) are all
  correct. Only ClaimAll is wrong. When the frontend sends a claim transaction,
  the contract doesn't recognize the opcode and throws error 130 ("Invalid
  incoming message"). The transaction bounces.

  There's also a secondary bug: the backend prematurely calls
  markChallengeClaimed() before the on-chain transaction completes. After the
  bounced tx, subsequent page loads see backendClaimed = true and hide the claim
   button entirely.

  Fixing both:

⏺ Two changes made:

  1. Root cause — contract.ts:10: Wrong OP_CLAIM_ALL opcode. Was 0xf9e43eb6,
  should be 0xf9dddb36 (matching the Tact ABI header 4192066358). Every claim
  transaction was unrecognized by the contract and bounced with error 130.
  CreateChallenge, AddFunds, and RefundUnclaimed opcodes were all correct — only
   ClaimAll was wrong.
  2. Secondary bug — verify.ts: Removed premature markChallengeClaimed() from
  the sign-proof route. The backend was marking challenges as claimed before the
   on-chain transaction even happened. After the bounced tx (caused by bug #1),
  subsequent page loads saw backendClaimed = true, set fullyReleased = true, and
   hid the claim button permanently. Also removed the backendClaimed dependency
  from fullyReleased in ChallengeDetail.tsx so on-chain claimedCount is the sole
   source of truth.
